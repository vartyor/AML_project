문제 정의 — 왜 GNN이 필요한가?

자금세탁은 단일 계좌의 이상 행동이 아니라 여러 계좌가 엮인 네트워크 패턴으로 발생합니다. 예를 들어 "대포통장 → 분산 송금 → 합치기(레이어링) → 통합 인출(인티그레이션)" 같은 구조는 계좌별 피처만 보는 XGBoost·LightGBM 단독 모델로는 포착되지 않습니다. 따라서 "계좌 간 연결 구조 자체를 학습하는 모델"이 필수였고, 이것이 GNN을 선택한 근본 이유입니다.

선택 기술후보 / 대안결정 근거
GraphSAGE GCN, GAT 귀납적(Inductive) 학습 방식 → 학습에 없던 신규 계좌도 즉시 임베딩 가능. 실시간 AML 탐지에 적합.
XGBoost (하이브리드) GNN 단독 분류, MLP GNN은 네트워크 구조, XGBoost는 비선형 피처 상호작용 학습에 강함 → 두 모델의 강점 결합. SHAP TreeExplainer로 해석도 용이.
SHAP (TreeExplainer) Feature Importance, LIME 금융 규제상 "왜 의심스러운가" 근거 제시가 필수. Shapley 값으로 피처별 기여도를 수학적으로 공정하게 배분.
LangChain + ChromaDB FAISS, Pinecone ChromaDB는 로컬 Persistent Client 지원 → 외부 API 없이 폐쇄망 환경에서도 동작. LangChain은 체인 구성의 표준화.
Ollama (llama3.1) OpenAI API, Anthropic API 금융 데이터는 외부 전송 자체가 리스크. 로컬 LLM으로 민감정보 유출 리스크 제거. 스트리밍 API 지원.
Neo4j (GraphRAG) NetworkX, JanusGraph Cypher로 "3-hop 최단 경로" 같은 그래프 질의를 선언적으로 표현 → 자금세탁 경로 탐색에 최적.
Streamlit FastAPI + React, Flask 신입 1인 개발 환경에서 Python만으로 데이터 앱 배포 가능. @st.cache_resource로 모델 재로딩 비용 제거.

TS-01SAR 보고서가 매번 다른 구조로 생성되는 문제
문제

같은 계좌를 여러 번 분석해도 LLM이 생성하는 SAR 섹션 순서·제목·마크다운 포맷이 매번 달라져 템플릿 파싱이 실패했습니다. 서비스화 관점에서 치명적이었습니다.

원인

① LLM 출력의 비결정성(기본 temperature), ② 모델이 ##II##를 **##II##**, ## II ##, ### ##II## 등으로 변형해서 출력, ③ Python str.format()이 LLM 출력 속 {var}를 키로 오인.

해결

3층 안정화 메커니즘을 구현. ① 생성 파라미터 고정(temperature=0.1, seed=42), ② _normalize_markers()로 마크다운 변형 사전 정규화, ③ _safe_section()으로 중괄호를 {{ }}로 이스케이프해 str.format() 충돌 차단.

코드
reporters/sar_template.py
def _normalize_markers(text: str) -> str:
    # **##II##** → ##II##  (앞뒤 마크다운 기호 제거)
    text = re.sub(r'[*`\[\]()]+\s*(##\w+##)\s*[*`\[\]()]+', r'\1', text)
    # ## II ## → ##II##  (내부 공백 제거)
    text = re.sub(r'##\s+(\w+)\s+##', r'##\1##', text)
    # ### ##II## → ##II##  (마크다운 헤더 제거)
    text = re.sub(r'^[ \t]*#{1,3}[ \t]+(##\w+##)', r'\1',
                  text, flags=re.MULTILINE)
    return text

def _safe_section(text: str) -> str:
    # LLM 출력의 { } → {{ }} 이스케이프 → str.format() 충돌 방지
    return text.replace("{", "{{").replace("}", "}}")

TS-03SHAP TreeExplainer가 학습된 XGBoost 모델에서 터지는 문제
문제

학습 후 저장한 XGBoost 모델을 불러와 shap.TreeExplainer(model)을 호출하면 ValueError: invalid literal for float()가 반복 발생.

원인

XGBoost 최신 버전에서 모델 설정의 base_score가 단일 float 대신 "[0.5]" 형태의 문자열 배열로 저장되는 내부 버그. SHAP가 이를 float으로 파싱하려다 실패.

해결

Explainer 초기화 전에 booster.save_config()로 JSON을 받아 base_score를 직접 float 변환 후 load_config()로 덮어쓰는 전처리를 추가. 추가로 TreeExplainer 실패 시 KernelExplainer로 자동 폴백하는 체인을 구성해 어떤 모델 버전에서도 SHAP가 동작하도록 했습니다.

코드
analysis/shap_analyzer.py
def _fix_base_score(self) -> xgb.Booster:
    booster = self.xgb_model.get_booster()
    try:
        config = json.loads(booster.save_config())
        b_score = config["learner"]["learner_model_param"]["base_score"]
        if isinstance(b_score, str):
            fixed = float(b_score.strip("[]"))        # "[0.5]" → 0.5
            config["learner"]["learner_model_param"]["base_score"] = str(fixed)
            booster.load_config(json.dumps(config))
    except Exception as exc:
        logger.warning("base_score 수정 중 오류 (무시): %s", exc)
    return booster

def build_explainer(self, X_input):
    try:
        self._explainer = shap.TreeExplainer(self._fix_base_score())
    except Exception:
        # 폴백: 느리지만 모든 모델에 적용 가능
        self._explainer = shap.KernelExplainer(
            self.xgb_model.predict_proba, X_input
        )
TS-04PaySim 그래프 희소성으로 인한 시각화 무의미 문제
문제

pyvis로 자금 흐름을 그려봤지만 PaySim 데이터의 평균 degree가 약 0.52 수준이라 대부분의 노드가 고립점으로 떠 있어 세탁 링(ring) 자체가 시각적으로 드러나지 않았습니다. 수사관이 "왜 이 계좌가 의심스러운가"를 납득할 수 없는 결정적 문제.

원인

PaySim 원본은 개별 거래 단위라 단일 거래 그래프로만 보면 대포통장·합치기 계좌가 서로 직접 연결되지 않음. "직접 거래는 없지만 같은 패턴을 공유하는 계좌"를 표현할 방법이 필요했습니다.

해결

이중 레이어 시각화를 도입. ① 실선(실거래) — edge_index BFS 2-hop, 화살표 방향 + 금액을 선 굵기로, ② 보라 점선(패턴 유사) — GNN 임베딩 공간의 코사인 KNN으로 같은 세탁 링 소속 가능성이 높은 계좌를 동시에 표시. 거래 금액은 np.expm1(log_amount)로 원래 ₩단위로 역변환.

코드
analysis/network_visualizer.py (핵심 발췌)
# 1) 실거래 레이어 — edge_index 기반 BFS 2-hop
for src, dst in bfs_edges(center_node, depth=2):
    amount = np.expm1(log_amount[(src, dst)])       # ₩로 역변환
    net.add_edge(src, dst, value=amount, arrows="to", color="#60a5fa")

# 2) 패턴 유사 레이어 — GNN 임베딩 코사인 KNN (핵심 성능 최적화)
#    embeddings: [N, 64]  →  cosine_sim: [N, N]  (벡터화 1회 연산)
embeddings_norm = embeddings / np.linalg.norm(embeddings, axis=1, keepdims=True)
cosine_sim = embeddings_norm @ embeddings_norm.T
top_k = np.argsort(-cosine_sim[center_node])[1:K+1]
for nbr in top_k:
    net.add_edge(center_node, nbr, color="#a78bfa", dashes=True)
TS-05Ollama LLM 타임아웃 — 응답이 시작되기도 전에 끊김
문제

llama3.1 8B로 SAR을 생성할 때 평균 30~60초가 걸리는데, 기본 requests.post(..., timeout=30)으로 매번 ReadTimeoutError가 발생하여 보고서 생성 자체가 실패했습니다.

원인

HTTP read timeout은 "전체 응답" 기준이라 대형 LLM에서는 부적합. 단순히 timeout을 늘리면 이번엔 "진짜 끊긴 경우"를 감지 못해 UX가 악화.

해결

Ollama의 스트리밍 API(stream=True)로 전환하고 timeout=(connect=10s, read=None)로 설정. 스트리밍 모드에서는 토큰 수신 때마다 read timeout이 초기화되므로 "첫 토큰은 10초 안에 와야 하지만, 이후 전체 응답 시간은 무제한"이라는 이상적인 타임아웃 구성이 됩니다. Streamlit st.write_stream과 결합해 실시간 출력도 확보.

코드
reporters/ai_report_generator.py
def stream_ai_report(json_data, model="llama3.1", ...) -> Generator:
    payload = {
        "model":   model,
        "prompt":  _build_prompt(json_data, rag_context, graph_context),
        "stream":  True,
        "options": {"temperature": 0.1, "seed": 42},  # 출력 일관성
    }
    # connect=10s, read=None(무제한)  — 스트리밍 타임아웃의 핵심
    with requests.post(ollama_url, json=payload, stream=True,
                       timeout=(10, None)) as resp:
        resp.raise_for_status()
        for raw_line in resp.iter_lines():
            token = json.loads(raw_line).get("response", "")
            if token: yield token
TS-06데이터 누수(Data Leakage) 4종 차단
problem

초기 모델의 Val AUC가 0.99+로 의심스러울 정도로 높게 나왔습니다. 실제 운영 데이터에서는 절대 재현되지 않을 "누수된" 성능이었습니다.

원인

누수 경로가 ① 시간 역전(랜덤 분할로 미래 거래가 train에 섞임), ② 라벨 기반 피처(fraud_send_ratio처럼 정답이 피처 계산에 들어감), ③ 스케일러 누수(StandardScaler.fit이 전체 노드로 수행), ④ 다중공선성(step과 hour_of_day 간 r≈0.97)으로 총 4가지가 동시에 존재.

해결

각 누수마다 개별 방어선을 구축. ① 계좌별 earliest_step 기준으로 시계열 정렬 후 70/10/20 분할, ② isFraud 기반 파생 피처 전량 삭제, ③ 스케일러를 train 인덱스만으로 fit 후 transform만 전체 적용, ④ hour_of_day_norm = (step % 24) / 23.0으로 직접 계산해 step과의 상관을 r≈0.03까지 낮춤. 재학습 결과 Val AUC가 현실적인 구간(0.87~0.92)으로 수렴.

TS-07Cypher avg(count()) 중첩 집계 오류 (GraphRAG Q4)
문제

"이 계좌가 평균적인 계좌보다 몇 배 더 많이 송금하는가(허브 계좌 지표)"를 구하려고 avg(count(r))를 썼더니 Neo4j가 Aggregation nested inside aggregation 오류를 반환.

원인

Cypher는 집계 함수의 중첩을 허용하지 않음. avg 안에 count를 바로 넣을 수 없고, 두 단계로 분리된 쿼리가 필요.

해결

3단계 쿼리로 분리: (1) 전체 엣지 수 ÷ 전체 노드 수로 평균 out-degree 선계산 → (2) 대상 계좌의 out-degree 계산 → (3) 둘의 비율(degree_ratio)을 산출. degree_ratio ≥ 3.0이면 "허브 계좌 의심: 평균의 3배 이상 송금" 경고를 SAR에 자동 삽입.

코드
knowledge/graph_rag.py — Q4 Cypher
// Step 1: 전체 평균 out-degree = 총 엣지 수 / 총 노드 수
MATCH (all_nodes:Account)
OPTIONAL MATCH (all_nodes)-[:SENT_TO]->()
WITH count(1) AS total_edges, count(DISTINCT all_nodes) AS total_nodes

// Step 2: 대상 계좌 out-degree 계산
MATCH (center:Account {node_idx: $node_idx})
OPTIONAL MATCH (center)-[r:SENT_TO]->(out:Account)
WITH total_edges, total_nodes, center, count(r) AS out_degree

// Step 3: 배율 산출 = out_degree / 전체 평균
RETURN
  CASE WHEN total_nodes > 0
    THEN round(toFloat(out_degree) /
               (toFloat(total_edges) / total_nodes) * 100) / 100.0
    ELSE 0.0
  END AS degree_ratio
